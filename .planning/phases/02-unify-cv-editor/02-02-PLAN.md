<objective>
Make the CV upload and storage pipeline produce robust JsonResume data by improving normalization heuristics and providing a migration path for existing freeform CVs.

Purpose: The current parsing prompt explicitly tells AI "Do NOT use or force any predefined schema" which produces freeform JSON. The `normalizeCvFieldNames` and `normalizeSectionNames` functions attempt to fix this but miss many edge cases. The result is CVs stored as pseudo-freeform data that the renderer struggles with. By making normalization more aggressive and comprehensive, we ensure most CVs become proper JsonResume that works flawlessly with the structured editor.
Output: Improved normalization utilities, updated upload pipeline, and a migration script for existing data.
</objective>

```typescript
export function detectCvFormat(cvJson: Record<string, any>): 'json-resume' | 'freeform' {
    const nonMetaKeys = Object.keys(cvJson).filter(k => k !== '__vh_tags' && !k.startsWith('__'));
    if (nonMetaKeys.length === 0) return 'json-resume';
    const knownCount = nonMetaKeys.filter(k => JSON_RESUME_KEYS.has(k)).length;
    const ratio = knownCount / nonMetaKeys.length;
    return ratio < 0.5 ? 'freeform' : 'json-resume';
}
```

Add a new function `compactFreeformToJsonResume` to `server/src/utils/cvNormalizer.ts` that takes freeform AI-parsed CV data and compacts it into a clean `JsonResumeSchema`.

The function should:

1. **Run existing normalization first** — call `normalizeSectionNames` and `normalizeCvFieldNames` on the input to handle the easy cases.

2. **Handle the 'Header Info' / 'basics' section** — Many freeform CVs have contact info under keys like 'Header Info', 'KONTAKT', 'PERSONALIEN'. Extract:
   - `name`: Look for fields tagged 'name' or keys matching `/^name$/i`
   - `label`: Look for 'title', 'headline', 'label' fields
   - `email`: Look for 'email' or patterns like `/\S+@\S+\.\S+/`
   - `phone`: Look for 'phone', 'tel', 'mobile' fields or patterns with `+` and digits
   - `url`: Look for 'website', 'url', 'linkedin', 'github' fields
   - `summary`: Look for 'summary', 'profile', 'objective', 'about' fields
   - `location.address`, `location.city`, `location.region`, `location.postalCode`, `location.countryCode`: Extract from location-related fields
   - `profiles`: Extract LinkedIn, GitHub, Xing, etc. as `{ network, url }` objects

3. **Handle work experience entries** — Freeform entries often use 'title', 'subtitle', 'dates', 'bullets'. Map to JsonResume:
   - `name` or `company` ← 'subtitle', 'company', 'employer', 'institution', 'organization'
   - `position` or `jobTitle` ← 'title', 'position', 'role', 'jobTitle'
   - `startDate` / `endDate` ← Parse 'dates' field. Split on common separators (`–`, `-`, `bis`, `to`, `/`). Handle formats like "01/2020 – 12/2023", "2020-2023", "Jan 2020 - Present". Use ISO format (YYYY-MM) where possible, keep original string if parsing fails.
   - `summary` ← 'summary', 'description' (as string)
   - `highlights` ← 'bullets', 'highlights', 'responsibilities', 'achievements' (as array of strings)
   - `location` ← 'location' field
   - `url` ← 'url' field

4. **Handle education entries** — Similar to work but map to education fields:
   - `institution` ← 'subtitle', 'institution', 'school', 'university'
   - `area` ← 'area', 'field', 'subject' or extract from title
   - `studyType` / `degree` ← 'title', 'degree', 'studyType', 'qualification'
   - `startDate` / `endDate` ← Same date parsing as work
   - `score` ← 'score', 'grade', 'gpa'
   - `courses` ← 'courses', 'subjects' (array)

5. **Handle skills** — Freeform skills come in many shapes:
   - Array of category objects with 'category'/'name' and 'content'/'keywords' → Map to `{ name: category, keywords: content.split(/,|;/) }`
   - Flat array of strings → Create a single skill entry `{ name: 'Skills', keywords: array }`
   - Single string with comma/semicolon separation → Split into keywords
   - Object with category keys → Convert each key to a skill entry

6. **Handle languages** — Common freeform patterns:
   - Array of objects with 'language'/'category' and 'fluency'/'content' → Map properly
   - Single string like "German (Native), English (Fluent)" → Parse with regex `/([^(]+)\(([^)]+)\)/g`
   - Array of strings → Try to split on dash, colon, or parenthesis

7. **Handle certificates, projects, awards, volunteer, interests, references, publications** — Apply similar heuristic mapping. For each:
   - Detect the section from normalized name
   - Map common field aliases to JsonResume field names
   - Handle both array-of-objects and flat-string formats

8. **Clean up the output**:
   - Remove empty arrays and empty objects
   - Remove null/undefined values
   - Ensure arrays actually contain items (filter out empty strings/nulls)
   - Keep `meta` if it exists, otherwise don't add it
   - Remove `__vh_tags` and any `__*` keys from the output (they're not part of JsonResume)

9. **Idempotency**: If the input already looks like valid JsonResume (most top-level keys are known JsonResume keys and structures match), return it with minimal changes (just cleanup).

The function signature:
```typescript

```

Add the function AFTER the existing `normalizeCvFieldNames` function in the file. Do NOT remove existing functions — they're still used by legacy code paths.

Write inline comments for each major section explaining the heuristic. Keep the code organized and readable — this is complex heuristic code that will need maintenance.
